<objective>
Lock down the API monolith authentication layer: remove the Kong header bypass from the authenticate middleware so every request requires a valid JWT, and restrict the registration schema to prevent ADMIN/DELIVERY role self-assignment.

Purpose: These are the two highest-severity security vulnerabilities. The Kong header bypass allows full identity impersonation without a token. The open role enum allows privilege escalation to ADMIN at registration.

Output: Hardened auth.ts middleware and restricted registerSchema in auth.controller.ts.
</objective>

  <name>Task 1: Remove Kong header bypass from authenticate middleware (D-01, D-02)</name>
  <files>services/api-monolith/src/shared/middleware/auth.ts</files>
  <action>
Modify the `authenticate()` function in `services/api-monolith/src/shared/middleware/auth.ts`:

1. Remove lines 17-24 entirely — the block that reads `x-user-id` and `x-user-role` headers and sets `req.user` from them. Per D-01, the Kong header shortcut must be fully removed.

2. The function should now ONLY authenticate via JWT:
   - Read `req.headers.authorization`
   - If missing or does not start with `'Bearer '`, return `res.status(401).json({ success: false, error: 'No token' })`
   - Otherwise, `jwt.verify(header.slice(7), env.JWT_ACCESS_SECRET)` and set `req.user`
   - On verify failure, return `res.status(401).json({ success: false, error: 'Invalid token' })`

3. Per D-02, Kong headers may still arrive on the request but are completely ignored by `authenticate()`. Do NOT read them at all. If someone wants them for logging, that is a separate concern outside this middleware.

4. Do NOT modify `optionalAuth` or `requireRole` — they are already correct.

The resulting `authenticate` function should be approximately 12 lines (header check + try/catch verify block).
