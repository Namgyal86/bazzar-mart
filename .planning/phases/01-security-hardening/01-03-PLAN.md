<objective>
Sanitize all error responses in the API monolith so production never leaks internal details, and configure Morgan logger to use the appropriate format per environment.

Purpose: The current global errorHandler and all inline catch blocks expose raw `err.message` (which can contain DB errors, file paths, stack traces) to clients. Morgan in `dev` mode logs request paths and query strings that may contain PII in production log aggregators.

Output: A shared `handleError` helper used by all controllers, a sanitized `errorHandler` middleware, and environment-conditional Morgan config.
</objective>

**Step 1: Update error.ts** (`services/api-monolith/src/shared/middleware/error.ts`):

1. Add `import { env } from '../../config/env';` at the top.

2. Create and export a `handleError` helper function per D-08:
```typescript
export function handleError(err: unknown, res: Response): void {
  const error = err instanceof Error ? err : new Error(String(err));
  console.error('[error]', error.message, error.stack);

  if ((err as any)?.name === 'ZodError') {
    res.status(400).json({ success: false, error: (err as any).errors });
    return;
  }

  const message = env.NODE_ENV === 'production'
    ? 'Internal server error'
    : error.message || 'Internal server error';

  res.status(500).json({ success: false, error: message });
}
```

3. Update the global `errorHandler` middleware per D-09 to apply the same production sanitization:
```typescript
export function errorHandler(err: Error, _req: Request, res: Response, _next: NextFunction): void {

  const message = env.NODE_ENV === 'production'
    ? 'Internal server error'
    : err.message || 'Internal server error';
  res.status(500).json({ success: false, error: message });
}
```

4. Keep `notFound` unchanged — per D-07, 404 responses preserve their user-facing messages.
